Getting Started

PushButton Engine is designed to be easy to start using. This chapter will give you a quick introduction to some of the tools we recommend for Flash development, talk about how to install and use the Engine Manager, and how to compile the Engine.

If you already have a favored development environment and don't need to rebuild the Engine from source, you can go to the Library folder, copy the provided SWCs into your class path, and start developing.

The rest of the chapters in this manual explain how to use the Engine.

Precompiled SWCs?

If you already have a Flash development environment set up, all you need to do is grab the precompiled SWCs from the Library folder and go.

If you are newer to Flash development, or need to compile the Engine from scratch, read the rest of this chapter for more information.

ActionScript or Flex?

The Engine will work in projects that only use ActionScript as well as projects that use the Flex SDK. You are free to do whatever best fits the needs of your project.

We include two basic demos/game templates. PBFlexlessEngineDemo is an example of how you will want to set up your ActionScript files for a basic game that does not use Flex. PBEngineDemo is the same, but using Flex. Both demos are included in the SDK download, in Projects/Demos.

The Tools

There are three tiers of support for development tools.

The first tier are tools that the Engine works with, and that we use to build the engine from scratch. They are supported by the Engine Manager, which automates a lot of project management tasks. If you will be doing development of new component libraries or modifying the engine you will want to use them. These tools are:

  • Adobe Flex Builder, an Eclipse-based IDE for ActionScript development.
  • Apache Ant, a command line make-like tool which along with the free Flex SDK can compile the Engine and applications.

The second tier are tools that will work with the Engine, but cannot rebuild it on their own. We provide documentation on how set them up. They are:

  • Adobe Flash, a visual tool for creating Flash content.
  • FlashDevelop, a free Windows-based ActionScript IDE.

The third tier is made up of tools that can work with the Engine, but that we have not tested with or written specific documentation for:

  • FDT, a 3rd party Eclipse-based ActionScript IDE.
  • Any other Flash development tool that supports using SWCs.

You may want to take a look at each of these tools before deciding what to use for your day to day development. Some of them are free, and the rest have time-limited trials. Choose one that you are comfortable with and that fits in your budget.

Installation For Second And Third Tier Tools

For FlashDevelop, read Jeremy Easoz's tutorial on Getting Started With FlashDevelop.

For Adobe Flash, follow the instructions in Set the Library path for ActionScript 3.0 files in the Flash CS4 documentation to add the SWCs in the Library folder from the SDK to your class path.

For any other tool, simply add the SWCs from the Library/ folder to your classpath, and you can start development immediately.

Most other IDEs do not have support for multiple projects, so using them with the PushButton Engine SDK can be difficult, since it uses many projects to keep things modular. However, editing any individual project is easy - create the project, point the source path of your project to the Source/ folder of the project, and include the Library/ folder in your classpath.

Installation For First Tier Tools

With the first tier tools, if you are already familiar with them and want to start using the Engine immediately, you can add the Library/ folder to your class path and start right away. However, if you are not as familiar with them, want to rebuild the engine from source, or want to develop component libraries for distribution, you should follow these steps in order to get set up with the Engine Manager.

First, go to the PushButton Engine download page and get the latest version of the engine. Extract the SDK somewhere that you'll be able to find it again later - the desktop is often a good choice if you don't already have a place for such things.

The PushButton Engine Manager is a small AIR application meant to simplify development of complex multiproject applications built around PushButton Engine. It provides several features:

  • Simplifies generating build files for multiple tools. Maintaining build files for Ant and Flex Builder by hand is guaranteed to result in inconsistencies, especially when hardcoded paths may be required (for instance, Flex Builder requires a hardcoded path to the root of the SDK). Using a tool simplifies all this.
  • Simplifies managing component packages. The Engine Manager allows you to package component libraries for distribution as well as download component packages and add them to your solutions.
  • Automates creating new projects. The Engine Manager automatically creates everything needed for a new project based on your choice of templates. This helps simplify creation of new component libraries or applications.

To get the SDK ready for use, double click on the ProjectManager.air file included in the root of the SDK to install the PushButton Manager. You may need to get the Adobe AIR runtime from http://get.adobe.com/air/ first.

The PushButton Manager is associated with .pbesln files. Once it has been installed, double click on PBE.pbesln in the root of the SDK. This will launch the PushButton Manager and let it know where you have installed the PushButton Engine SDK. A wizard will automatically start and prompt you to create build files.

It will also prompt you to specify the location of your Flex SDK. This should be set to the root of the Flex SDK path which has folders called ant, asdoc, bin, and several others. If you have installed Flex Builder (see step 2), then the Flex SDK is located under the sdks subfolder of Flex Builder. If you have installed just the Flex SDK, select its location. The Engine Manager will check if you have a valid SDK selected. (If no wizard appears, you can click the "Generate Build Files..." option in the left hand column.)

At this point you can close the PushButton Manager and move on to the next step.

Setting Up Flex Builder

Note: This section applies whether you are using Flex Builder, or have installed Eclipse and added the Flex Builder Plugin. For simplicity, we will only refer to Flex Builder.

If you haven't yet, go to the Adobe Flex Builder download page and download and install Flex Builder.

Open Flex Builder, go to File -> Choose Workspace -> Other... and choose Build/FlexBuilder in the SDK folder when it asks for a path. Hit OK and you will get a nice new workspace with a few key settings already prep'ed for you.

You may want to disable automatic builds (under Project -> Build Automatically), as they tend to slow Flex Builder down dramatically.

Once this is done, right click on the project explorer, choose Import..., then General -> Existing Projects Into Workspace.. and hit Next. Select Build/FlexBuilder as the root directory. You may have to change focus away from that text box (by clicking on another text box or hitting tab) for the list of projects below to update. Make sure all the projects in the list on the bottom part of the dialog are checked and click Finish to import them.

The workspace should refresh, and everything should start compiling automatically. If a build doesn't happen automatically, you can press CTRL-B to trigger one.

Now you can build, browse project source code, and run/debug the demo applications.

Setting Up Ant and the Free Flex SDK

To do command line builds with the free Flex SDK and Ant, you will need to have Ant installed and in your path. Then you can go to Build/Ant and run the batch files to perform common build tasks (for instance, run Build.bat or Build.command to compile the engine).

To get Ant, go to ant.apache.org and get the appropriate version. You may want to use an installer like WinAnt if you're on Windows, as Ant will need to be in your path.

You can build specific projects by running Ant yourself from the Build/Ant folder via the command line. Do this by typing ant projectname. There are several other convenience targets; read the comments at the top of Build/Ant/build.xml for details, but they include:

  • clean - To clean all build artifacts.
  • clean.projectname - To clean build artifacts from a specific project.
  • -DDEBUG=true - Pass this parameter to do a debug build.

SDK Layout

Now that you are able to build, let's talk about how files in the SDK are organized.

  • Build/ - Contains project and build files for various development tools generated by the project manager. There is a subdirectory for each support development tool.
    • FlexBuilder/ - Contains FlexBuilder related build files.
    • FlexSDK/ - Contains Ant build files to build using the free Flex SDK.
  • Documentation/ - Contains this manual, which covers the core engine released by PushButton Labs. Third-party components' documentation is in that component's project folder (see below).
  • Libraries/ - Compiled SWCs from each library project are stored here. This is in the classpath for every project so they can access components.
  • Projects/ - One or more projects are stored in here. The project generator scans for directories with .pbeproj files when it is generating build files. Note they can be nested as deep as you like to keep things organized, and relative location doesn't matter.
    • Demos/ - Contains projects that demonstrate component libraries.
    • Engine/ - Contains projects that implement core engine features.
  • Templates/ - Stores template projects for the Engine Manager's use. Templates are stored as ZIPs and their format is discussed later.

Project Layout

Each project contained in the Projects folder has the same file structure. This lets the Engine Manager automatically generate build files, and keeps development very modular so it's easy to share components with others.

Projects cannot be nested within one another. The name of the project is the name of its folder. At its root, every project contains a .pbeproj file named after the project, for instance the PBEngine project contains PBEngine.pbeproj. The .pbeproj is a small XML file that describes the project, what it is, who made it, and what its dependencies are.

Projects have a type, specified in their .pbeproj file. The type can be one of:

  • Library - indicating that the project produces a SWC, named after the project. So for the RenderingTwoD project, RenderingTwoD.swc.
  • WebApplication - indicating that the project should build a SWF, in that project's Output/ subdirectory. The Source/ folder should contain a .mxml or .as file named after the application, which is what will be compiled to produce the SWF. For the PBEngineDemo, Source/PBEngineDemo.mxml is compiled to produce Output/PBEngineDemo.swf. For PBFlexlessEngineDemo, Source/PBFlexlessEngineDemo.as is compiled to produce Output/PBFlexlessEngineDemo.swf.

The type influences the directories that are used. For a library, the structure looks like this:

  • LibraryName.pbeproj - The project description file for this library. Note it will be named after the library.
  • Documentation/ - Contains an index.html that contains or links to documentation for that library.
  • Source/ - Source files for the library. Every file in this folder or any subdirectory will be included in the library.

The results of building the library are stored as a SWC in the root Libraries folder in the SDK.

For an application, the structure is a little more complex:

  • Assets/ - The assets used by this project.
    • Images/ - Image files (jpg, png, gif, etc).
    • Levels/ - Level XML files.
    • Sounds/ - Sound files (mp3).
  • Output/ - The SWF output from the project.
  • Source/ - Source code for the project.
